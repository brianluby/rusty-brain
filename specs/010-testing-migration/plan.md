# Implementation Plan: Testing & Quality + Migration & Backwards Compatibility

**Branch**: `010-testing-migration` | **Date**: 2026-03-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/010-testing-migration/spec.md`

## Summary

Achieve comprehensive test coverage across all 7 workspace crates, add TypeScript-compatible `.mv2` migration testing, enforce performance benchmarks (≥2× TypeScript), introduce fuzz testing harnesses, and enhance CI gates. This is a testing-and-validation feature with no new runtime functionality beyond legacy path detection.

## Technical Context

**Language/Version**: Rust stable, edition 2024, MSRV 1.85.0
**Primary Dependencies**: memvid-core (pinned rev `fbddef4`), criterion 0.5 (benchmarks), cargo-fuzz/libFuzzer (fuzz testing), serde/serde_json (fixture parsing), tempfile (test isolation), assert_cmd/predicates (CLI tests)
**Storage**: `.mv2` files via memvid-core (read/write compatibility), `tests/fixtures/` (committed test data)
**Testing**: `cargo test` (unit + integration), `cargo bench` (criterion), `cargo fuzz` (libFuzzer)
**Target Platform**: macOS (dev), Linux (CI via GitHub Actions)
**Project Type**: Rust workspace (7 crates)
**Performance Goals**: ≥2× faster than TypeScript on query latency, compression throughput, and startup time
**Constraints**: 60-second minimum fuzz duration per harness in CI; ±0.01 score tolerance for compatibility; 5-second stale lock recovery timeout
**Scale/Scope**: ~250 existing tests → target ~400+ tests; 3 fuzz harnesses; 6 benchmark scenarios; committed fixture files

## Implementation Phases

### Phase 1: Fix Existing Failures + Unit Test Gaps (FR-001, FR-003)

**Goal**: Green baseline — fix 2 failing CLI tests, add unit tests to hooks (18 pub fns, 0 unit tests) and opencode (22 pub fns, 0 unit tests), enhance compression entry point coverage.

**Deliverables**:
- Fix `test_find_json_output_valid` and `test_find_type_filter_applies_before_final_limit` in CLI
- Unit tests for all public functions in hooks crate (`should_process`, `resolve_memory_path`, `open_mind`, `format_system_message`, all `handle_*` functions)
- Unit tests for all public functions in opencode crate (`handle_with_failopen`, `mind_tool_with_failopen`, sidecar API, chat/tool hooks)
- Unit tests for `compress()` entry point in compression crate (beyond panic recovery)
- Additional platform integration tests (expand from 1 to 3+ test files)

**Exit criteria**: `cargo test --workspace` passes with 0 failures; every `pub fn` across all 7 crates has ≥1 test

### Phase 2: TypeScript Compatibility Tests (FR-006, FR-009, FR-010, FR-011, FR-013)

**Goal**: Verify drop-in replacement capability against TypeScript-generated fixtures.

**Deliverables**:
- Generate and commit TypeScript `.mv2` fixtures to `tests/fixtures/`
- `expected_results.json` with reference search results, timeline, stats
- Compatibility test module (`crates/core/tests/compatibility/`) that loads fixtures and asserts identical results (±0.01 score tolerance)
- Environment variable compatibility tests for all 6 env vars
- Directory layout assertion tests (`.agent-brain/` structure matches TypeScript)

**Exit criteria**: All compatibility tests pass; search results match TypeScript within tolerance

### Phase 3: Legacy Path Detection (FR-012)

**Goal**: Detect `.claude/mind.mv2` and suggest migration.

**Deliverables**:
- `detect_legacy_path()` function in hooks bootstrap
- Structured `Diagnostic::Warning` output
- Unit and integration tests covering: legacy-only, both-exist, normal-only scenarios

**Exit criteria**: Legacy detection tests pass; diagnostic output is structured and machine-parseable

### Phase 4: Concurrency Tests (FR-004)

**Goal**: Validate parallel writer safety and stale lock recovery.

**Deliverables**:
- Extend `crates/core/tests/integration/mind_concurrent.rs` with 4, 8, 16 writer tests
- Add stale lock recovery test (verify recovery within 5 seconds)
- Add reader-during-write test (verify no partial reads)

**Exit criteria**: SC-003 — 4+ parallel writers, 100% success rate, no data loss

### Phase 5: Performance Benchmarks (FR-005, FR-008)

**Goal**: Prove ≥2× performance advantage and gate CI on regressions.

**Deliverables**:
- Capture TypeScript baselines in `tests/fixtures/ts_baselines.json`
- Enhance existing benchmarks with baseline comparison logic
- Add startup time benchmark (`crates/cli/benches/`)
- Benchmark report summarizing Rust vs TypeScript for each metric

**Exit criteria**: SC-004 — all metrics ≥2× faster than TypeScript

### Phase 6: Fuzz Testing (FR-007)

**Goal**: Exercise input boundaries with fuzz-generated data.

**Deliverables**:
- `crates/compression/fuzz/` — compression input fuzzer
- `crates/hooks/fuzz/` — hook JSON parsing fuzzer
- `crates/core/fuzz/` — search query parsing fuzzer
- Seed corpora for each harness
- Regression test generation from discovered crashes

**Exit criteria**: SC-006 — 60 seconds per harness, no panics

### Phase 7: CI Enhancement (FR-008, FR-014)

**Goal**: Gate all quality checks in CI.

**Deliverables**:
- Update `.github/workflows/ci.yml` with:
  - Benchmark regression step (fail on <2× vs baseline)
  - Fuzz smoke test step (60s per harness)
  - Coverage report step (optional, informational)
- Update `plugin.json` hook paths to Rust binaries (FR-014)

**Exit criteria**: SC-007 — all CI gates pass; SC-008 — plugin.json points to Rust binaries
